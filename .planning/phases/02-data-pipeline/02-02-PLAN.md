  <action>
Create `src/main/pipeline/discovery.ts` with two exported functions.

**`discoverSessions(baseDir?: string): Promise<SessionMetadata[]>`**

1. Default `baseDir` to `join(homedir(), '.claude', 'projects')` using `os.homedir()` and `path.join`.
2. Read the baseDir with `readdir`. If it fails (directory doesn't exist), return an empty array (don't throw).
3. For each entry in baseDir:
   - `stat` the entry. If not a directory, skip.
   - This is a project folder (e.g., "D--Code-PromptPlayer").
   - Read the project folder contents.
   - For each `.jsonl` file in the project folder (UUID-named files at the root level only -- do NOT recurse into subdirectories like `subagents/`):
     - Call `extractSessionMetadata(filePath, projectFolder)` to get metadata.
4. Return all SessionMetadata objects. Do NOT filter or sort -- the renderer will handle grouping/sorting.
5. Handle errors gracefully: if a project folder can't be read, skip it. If a file can't be scanned, return a SessionMetadata with `parseError` set.

**`extractSessionMetadata(filePath: string, projectFolder: string): Promise<SessionMetadata>`**

This does a FAST scan -- reads only the first ~50 lines per file to extract metadata without full parsing. This keeps discovery snappy even with hundreds of files.

1. Use `createReadStream` + `createInterface` (readline) with `encoding: 'utf-8'`.
2. Read up to 50 lines. Track:
   - `firstTimestamp`: earliest `timestamp` field encountered
   - `firstUserMessage`: first meaningful user message snippet (see strategy below)
   - `messageCount`: count of user + assistant type lines seen in the first 50 lines (this is an APPROXIMATION -- it won't be exact for large files, but it's fast)
3. Strategy for first user message snippet (from research):
   - Find the first line where `type === 'user'` AND `isMeta` is falsy
   - If `message.content` is a string: check it does NOT start with `<command-name>`, `<local-command`, `<command-message>`, `<execution_context>`, or `<objective>`. If clean, take first 150 chars as snippet.
   - If `message.content` is an array: find first `text` block that doesn't start with those XML prefixes. Take first 150 chars.
   - If no clean user message found in first 50 lines, `firstUserMessage` stays null.
4. On any read error (file unreadable, permission denied, etc.), return SessionMetadata with `parseError` set to the error message. Per locked decision: "Unparseable JSONL files appear in browse list with an error badge (not silently excluded)."
5. The `sessionId` is the filename without `.jsonl` extension.
6. Abort readline after 50 lines by calling `rl.close()` and breaking out of the loop.

Import `SessionMetadata` from `./types` (created in Plan 01). If Plan 01 hasn't run yet (parallel wave), define a local compatible interface and add a TODO comment to switch to the shared import once types.ts exists. The executor running this plan should check if types.ts exists first and import from it if available.

Use `import { readdir, stat } from 'fs/promises'` for async directory operations. Use `createReadStream` from `fs` and `createInterface` from `readline` for the streaming line reader.

This module persists sessions that the user adds to a presentation. Per locked decision: "Only sessions the user adds to a presentation get copied into app-local storage."

**Interface `StoredSession`:**
```typescript
interface StoredSession {
  sessionId: string;
  projectFolder: string;
  originalFilePath: string;
  messages: ParsedMessage[];  // Full parsed + stitched + classified messages
  metadata: SessionMetadata;
  addedAt: number;            // Date.now() when added to presentation
}
```

Import `ParsedMessage` and `SessionMetadata` from `../pipeline/types` (same parallel-wave note as Task 1 -- use local interface if types.ts doesn't exist yet).

**Exported functions:**

1. `getStoredSessions(): StoredSession[]` -- Read and parse `sessions.json` from `app.getPath('userData')`. Return empty array on any error (file missing, corrupt JSON).

2. `saveStoredSession(session: StoredSession): void` -- Load existing sessions, append or replace (match by sessionId), write back to `sessions.json`. Use `mkdirSync` with `{ recursive: true }` before writing (ensure userData dir exists). Use `writeFileSync` with `JSON.stringify(sessions, null, 2)` for readable JSON.

3. `removeStoredSession(sessionId: string): void` -- Load existing sessions, filter out the one with matching sessionId, write back.

4. `loadStoredSessions(): StoredSession[]` -- Alias for getStoredSessions (for API naming consistency with Phase 1 pattern).

Storage path: `join(app.getPath('userData'), 'sessions.json')`

This is a simple synchronous JSON file store. The data volumes are small (a presentation might have 5-15 sessions). No need for SQLite or async I/O here -- matches the Phase 1 pattern exactly.
