## Stitcher orphan bug (in progress)
The chain walking still produces high orphan counts (84/97 in one session, 53/59 in another). Only ~13 messages get chain-walked; the rest become orphans sorted by timestamp.

### Root cause investigation done
Ran analysis scripts on real JSONL files. Found:
- Claude Code JSONL chain: `user → assistant → system → user → ...`
- System lines (turn_duration metadata) sit between every assistant→user pair
- Parser correctly filters system lines but this breaks the parentUuid chain
- Added filteredUuidRedirects to bridge the gaps, but orphans are still high

### Likely remaining issue
**The `childOf` Map in stitcher.ts overwrites entries.** When multiple messages resolve to the same parentUuid after redirect resolution, only the LAST one stored in the Map survives. The chain walk follows that one child, and all other children of the same parent become orphans.

This would happen if:
1. After redirect resolution, multiple user messages end up pointing to the same assistant message as parent
2. Or the assistant turn reassembly creates uuid mappings that don't align with the chain

### How to verify
Write a script that checks: after redirect resolution, how many messages share the same parentUuid? If many do, that confirms the Map overwrite theory.

### Possible fix
Instead of `childOf: Map<string, ParsedMessage>` (one child per parent), use `childOf: Map<string, ParsedMessage[]>` and pick the right child during chain walking (probably the one with the earliest timestamp that hasn't been visited).
