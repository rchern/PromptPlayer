<task type="auto">
  <name>Task 2: JSONL parser with assistant turn reassembly</name>
  <files>src/main/pipeline/parser.ts</files>
  <action>
Create `src/main/pipeline/parser.ts` that exports `parseJSONLFile(filePath: string): Promise<ParseResult>`.

Implementation steps:
1. Use Node.js `createReadStream` (with `encoding: 'utf-8'` -- critical for Windows) + `createInterface` from readline to read line by line.
2. For each line: try `JSON.parse(line)`. On failure, record a `ParseError` with line number and skip.
3. Skip empty lines (blank or whitespace-only).
4. Filter out non-message line types: skip `file-history-snapshot`, `queue-operation`, and `progress` lines entirely. Only keep `user`, `assistant`, and `system` lines.
5. For `system` lines: skip them (they contain turn_duration metadata, not conversation content). We do NOT need them in the message model.
6. For `user` lines: create a `ParsedMessage` directly. Normalize content:
   - If `message.content` is a string, wrap it as `[{ type: 'text', text: content }]`
   - If `message.content` is an array, use it directly as `ContentBlock[]`
   - Set `isMeta` from the line's `isMeta` field (default false)
   - Set `toolVisibility` to null unless the content contains a `tool_result` block (handle in classifier later)
7. For `assistant` lines: do NOT create individual ParsedMessages. Instead, collect them for reassembly.
8. After reading all lines, reassemble assistant turns:
   - Group assistant lines by `requestId`
   - Within each group, preserve insertion order (they come in chain order in the file)
   - Merge content blocks: `group.flatMap(line => line.message.content)` (each line has exactly one content block in its array)
   - The reassembled turn's `parentUuid` = first line's `parentUuid`
   - The reassembled turn's `uuid` = last line's `uuid` (so children can link to it)
   - The reassembled turn's `timestamp` = first line's `timestamp`
   - The reassembled turn's `requestId` = the shared requestId
9. Combine user ParsedMessages and reassembled assistant turns. Do NOT sort yet -- that's the stitcher's job.
10. Return `{ messages, errors, totalLines }`.

Important edge cases from research:
- User message content can be a string OR an array -- always normalize to array
- Assistant lines each have exactly one content block but it's wrapped in an array
- isMeta user messages should be included in parse output but marked `isMeta: true` (filtering happens in stitcher/renderer)
- Lines with string content starting with `<command-name>`, `<local-command`, or `<command-message>` are command messages -- include but mark appropriately (the stitcher or renderer will filter these)
  </action>
  <verify>Run `npx tsc --noEmit` from the project root. No type errors. Manually verify the function signature exports correctly by checking that `import { parseJSONLFile } from './parser'` would resolve.</verify>
  <done>parseJSONLFile reads any JSONL file, skips malformed lines (recording errors), normalizes user content, reassembles split assistant turns by requestId, and returns a ParseResult with all messages and error info.</done>
</task>

  <action>
**classifier.ts** -- Create `src/main/pipeline/classifier.ts` with:

1. `PLUMBING_TOOLS` constant Set: `Read`, `Grep`, `Glob`, `Write`, `Edit`, `Bash` (per user's locked decision)
2. `NARRATIVE_TOOLS` constant Set: `AskUserQuestion`, `TaskCreate`, `TaskUpdate`, `TaskList` (per user's locked decision)
3. Export `classifyToolCall(toolName: string): ToolVisibility` -- returns 'plumbing', 'narrative', or 'unknown'
4. Export `classifyMessage(message: ParsedMessage): ParsedMessage` -- examines contentBlocks:
   - If any block is `tool_use`, classify by the tool's name
   - If any block is `tool_result`, the classification will be applied later when pairing with tool_use (set to null for now -- the stitcher pairs them)
   - If message has both text and tool_use blocks, the message gets the tool's classification (text portion visibility is handled by the renderer per locked decision: "text portion is still shown even when tool calls are hidden")
   - Thinking blocks: set toolVisibility to 'plumbing' (hidden by default per locked decision)
   - Messages with only text blocks: toolVisibility stays null (always shown)
5. Export `pairToolResults(messages: ParsedMessage[]): ParsedMessage[]` -- after stitching, walk the ordered messages and for each `tool_result` block, find the matching `tool_use` by `tool_use_id` and copy its classification to the tool_result message's `toolVisibility`. Per locked decision: "tool_result inherits the same classification as its paired tool_use."

**stitcher.ts** -- Create `src/main/pipeline/stitcher.ts` with:

1. Export `stitchConversation(messages: ParsedMessage[]): StitchedSession`
2. Build a Map of `uuid -> ParsedMessage` for all messages
3. Build a Map of `parentUuid -> ParsedMessage` (child lookup: given a uuid, find the message whose parentUuid matches it)
4. Find the root message: the one with `parentUuid === null`
5. Walk the chain from root: `current = root`, then repeatedly `current = childOf.get(current.uuid)` until no child exists
6. During the walk:
   - If `current.isSidechain === true`, skip it (exclude from output) but increment sidechainCount
   - Otherwise, add to the ordered array
7. Handle orphaned messages (broken parentUuid chain):
   - After chain walk, check if any messages were NOT visited
   - Orphans: messages whose parentUuid references a uuid NOT in the message set
   - Per locked decision: "Orphaned messages included via best-effort ordering (by timestamp)"
   - Sort orphans by timestamp and append them to the end of the ordered array
   - Track orphanedCount
8. If no root found (no null parentUuid), fall back to ordering ALL messages by timestamp
9. After ordering, run `classifyMessage` on each message, then run `pairToolResults` on the full ordered array
10. Return `{ messages: orderedAndClassified, orphanedCount, sidechainCount }`
