<blockers>
**ACTIVE: Missing user messages after tool rejections**

User reported that their typed response after rejecting an AskUserQuestion doesn't appear in the Player. Investigation traced the issue to the stitcher's `childOf` Map:

**Root cause hypothesis:** The stitcher builds `childOf` as `Map<parentUuid, child>` (line 54-58 of stitcher.ts). When multiple messages share the same parentUuid (after redirect resolution), the Map overwrites — only the LAST child survives. The chain walk (line 82: `current = childOf.get(current.uuid)`) then skips the overwritten message.

**Evidence from JSONL analysis (this session, lines 443-450):**
- L443: user (tool_result rejection), uuid=9b7d1d7b
- L444-445: assistant thinking+text (reassembled), uuid=3f55c8ac, parent=9b7d1d7b
- L446: system, uuid=29133a49, parent=3f55c8ac (SKIPPED, creates redirect)
- L448: user "We'll use this as an example!", uuid=94c6ae4d, parent=29133a49 → redirected to 3f55c8ac

L448's parent resolves to 3f55c8ac (the reassembled assistant). The childOf entry for 3f55c8ac should point to L448, enabling the chain walk. This specific case SHOULD work because there's only one child for that parent.

**But** in the user's original session (discuss-phase 4), there may be multiple children competing for the same parentUuid, causing the overwrite. This is the existing stitcher bug from STATE.md: "The childOf map may be losing entries when multiple messages share the same parentUuid (Map overwrites)."

**Fix approach:** Change `childOf` from `Map<string, ParsedMessage>` to `Map<string, ParsedMessage[]>` and walk ALL children, or use a different chain resolution strategy. This is a Phase 2 (stitcher) fix, not a Phase 4 fix.

**Alternative:** The missing message might also be a `queue-operation` type (see STATE.md blocker about queue-operation messages not being parsed). Need to check the original session's JSONL to confirm.
</blockers>
