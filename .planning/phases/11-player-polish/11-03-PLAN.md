1. **Change elapsedMs computation from cross-step to within-step.** Remove the `prevNavTimestamp` / `prevNavSessionId` tracking variables entirely. Instead, compute elapsedMs directly within each navigation step as the time from user message to assistant response:

Replace the loop body (lines ~118-137) that currently does:
```typescript

let elapsedMs: number | null = null
if (prevNavSessionId === ref.sessionId && prevNavTimestamp && currTimestamp) {
  elapsedMs = computeElapsedMs(prevNavTimestamp, currTimestamp)

prevNavTimestamp = currTimestamp
prevNavSessionId = ref.sessionId
```

With:
```typescript

const elapsedMs = computeElapsedMs(
  navStep.userMessage?.timestamp ?? null,
  navStep.assistantMessage?.timestamp ?? null
)
steps.push({ type: 'navigation', step: navStep, sessionId: ref.sessionId, sectionId: section.id, elapsedMs })
```

Delete the `prevNavTimestamp` and `prevNavSessionId` variable declarations (lines ~44-45) and the session-boundary reset (lines ~113-115) since they are no longer needed.

2. **Fix session duration lastTimestamp to prefer assistant timestamp.** Change `getStepTimestamp()` to prefer `assistantMessage?.timestamp` over `userMessage?.timestamp`:

```typescript
function getStepTimestamp(step: NavigationStep): string | null {
  return step.assistantMessage?.timestamp ?? step.userMessage?.timestamp ?? null
}
```

This ensures session duration measures from first user message to last assistant response (since `getStepTimestamp` is used for both first and last steps, and the first step will fall back to userMessage timestamp when assistantMessage is null -- but NavigationSteps always have an assistantMessage for the first step, so the first timestamp will be the assistant's, which is fine because the very first step's user timestamp would be the absolute earliest anyway only if present). Actually, to be precise: for session duration, the first timestamp should be the earliest timestamp and the last should be the latest. Since assistant messages come AFTER user messages chronologically, preferring assistant for the last step gives the correct endpoint. For the first step, if we prefer assistant, we might skip the user's earlier timestamp. To handle this correctly:

Change `getStepTimestamp` into two functions:
```typescript
/** Earliest timestamp for a step (prefers user message as it comes first). */
function getStepFirstTimestamp(step: NavigationStep): string | null {
  return step.userMessage?.timestamp ?? step.assistantMessage?.timestamp ?? null
}

/** Latest timestamp for a step (prefers assistant message as it comes last). */
function getStepLastTimestamp(step: NavigationStep): string | null {
  return step.assistantMessage?.timestamp ?? step.userMessage?.timestamp ?? null
}
```

Then update session duration computation (lines ~88-93) to use:
```typescript
const firstTimestamp = getStepFirstTimestamp(navSteps[0])
const lastTimestamp = getStepLastTimestamp(navSteps[navSteps.length - 1])
```

This gives the widest accurate session duration span: from the earliest user prompt to the latest assistant response.

<task type="auto">
  <name>Task 2: Fix usePlayerTheme async cleanup race condition</name>
  <files>src/renderer/src/hooks/usePlayerTheme.ts</files>
  <action>
The root cause: a single useEffect handles both theme application and unmount restoration. Its cleanup dispatches an uncancellable async `getTheme()` call. In React StrictMode (mount -> unmount -> remount), the cleanup's async callback resolves AFTER the remount's effect sets the correct theme, overwriting it with the system theme.

This same race also causes the toggle to appear broken (Test 7): toggling updates the store, which triggers a re-render, which re-runs the effect. But the previous effect's cleanup async resolves after the new effect sets the toggled theme, clobbering it.

Replace the current single useEffect with two separate effects:

**Effect 1: Apply theme (runs on effectiveTheme changes)**
```typescript
// Apply effective theme to document element
useEffect(() => {
  if (!presentation) return
  document.documentElement.setAttribute('data-theme', effectiveTheme)
}, [presentation, effectiveTheme])
```
No cleanup in this effect. It only sets the attribute when effectiveTheme changes.

**Effect 2: Restore system theme on unmount only (runs once)**
```typescript
// Restore system theme when Player unmounts (leaving presentation)
useEffect(() => {
  if (!presentation) return

  return (): void => {
    // Restore system theme on true unmount
    window.electronAPI.getTheme().then((isDark) => {
      document.documentElement.setAttribute('data-theme', isDark ? 'dark' : 'light')
    })
  }
  // eslint-disable-next-line react-hooks/exhaustive-deps
}, []) // Empty deps = only runs cleanup on true unmount, not on re-renders
```

Wait -- empty deps `[]` won't capture the current presentation state. And in StrictMode, even `[]` effects run mount->unmount->remount. The async cleanup from the first unmount will still race.

Better approach: Use a ref-based cancellation flag in the unmount effect:

```typescript
import { useEffect, useRef } from 'react'

export function usePlayerTheme(): { effectiveTheme: 'light' | 'dark'; isDark: boolean } {
  const presentation = usePlaybackStore((s) => s.presentation)
  const themeOverride = usePlaybackStore((s) => s.themeOverride)
  const isDarkMode = useAppStore((s) => s.isDarkMode)
  const isMountedRef = useRef(true)

  // Resolve effective theme (same as current)
  let effectiveTheme: 'light' | 'dark' = 'light'
  if (presentation) {
    if (themeOverride) {
      effectiveTheme = themeOverride
    } else if (presentation.settings.theme === 'system') {
      effectiveTheme = isDarkMode ? 'dark' : 'light'
    } else {
      effectiveTheme = presentation.settings.theme
    }
  }

  // Effect 1: Apply theme immediately when effectiveTheme changes
  useEffect(() => {
    if (!presentation) return
    document.documentElement.setAttribute('data-theme', effectiveTheme)
  }, [presentation, effectiveTheme])

  // Effect 2: Restore system theme on unmount with cancellation guard
  useEffect(() => {
    isMountedRef.current = true

    return (): void => {
      isMountedRef.current = false
      // Delay restoration slightly so that if this is a StrictMode remount,
      // the remount's isMountedRef.current = true runs first and the
      // async callback checks the ref before writing.
      // Use a microtask delay (queueMicrotask or setTimeout(0)) so the
      // remount effect has a chance to set isMountedRef.current = true.
      setTimeout(() => {
        if (!isMountedRef.current) {
          window.electronAPI.getTheme().then((isDark) => {
            if (!isMountedRef.current) {
              document.documentElement.setAttribute('data-theme', isDark ? 'dark' : 'light')
            }
          })
        }
      }, 0)
    }
  }, [])

  return { effectiveTheme, isDark: effectiveTheme === 'dark' }
}
```

Key behavioral changes:
- Theme application is a simple synchronous setAttribute, no cleanup needed
- Unmount restoration uses isMountedRef to detect StrictMode remounts vs true unmounts
- setTimeout(0) ensures the remount's `isMountedRef.current = true` executes before the cleanup checks the flag
- Double-check inside the `.then()` callback guards against a real unmount followed by a quick remount
- The toggle now works because Effect 1 has no cleanup that could race -- it just sets the attribute
